# ADR-018: Hosted Streamable HTTP Public Preflight Mode

- Status: Accepted
- Date: 2026-07-03
- Implementation status: Planned

## Context

The repository already implements a local stdio upstream server and a runtime
Gateway path for managed environments. PlayMCP/Kakao registration has a
different shape: PlayMCP needs a public Remote MCP endpoint, and public users
should receive pre-use decision support before they connect another MCP.

Opening the full runtime Gateway surface to public PlayMCP users would expose
target-call, approval, audit, rescan, and operator concepts that do not belong in
the public listing. It would also blur the claim boundary between static
preflight and managed runtime enforcement.

## Decision

Add a distinct hosted mode named `public-preflight`.

`public-preflight` is an inbound Streamable HTTP MCP server intended for
PlayMCP/Kakao registration. It exposes only:

- `gateway_search_playmcp`
- `gateway_preflight_mcp`
- `gateway_explain_mcp_risk`

It must not expose:

- `gateway_health` as an MCP tool; use HTTP `/healthz` instead.
- `gateway_call_tool`
- `gateway_list_exposed_tools`
- `gateway_request_approval`
- target registry, rescan, diff, audit read, or operator tools.
- dynamic target aliases.

The public server is stateless from the target-MCP perspective: it does not
spawn, register, or call target MCPs. It returns inventory-based decision aids,
risk labels, policy recommendations, and handoff text.

## Required Hosted Controls

- single public MCP endpoint, default `/mcp`.
- HTTPS before public registration.
- `Origin` validation when present.
- POST requests accept one JSON-RPC message and require an `Accept` header that
  includes `application/json` and `text/event-stream`.
- JSON-RPC notifications/responses return `202 Accepted` when accepted.
- JSON-RPC requests return either `application/json` or `text/event-stream`.
- GET either opens an SSE stream or returns `405 Method Not Allowed`.
- DELETE returns `405 Method Not Allowed` while the public server remains
  stateless.
- `MCP-Protocol-Version` is validated; unsupported values return `400 Bad
  Request`.
- `Mcp-Session-Id` is not issued by default. If a future stateful mode issues
  one, all follow-up requests must validate it.
- body-size and query-length limits.
- rate limiting or abuse throttling before public listing.
- `/healthz` that does not reveal tenant IDs or private target metadata.
- logs that avoid raw credentials, private target endpoints, and raw audit
  payloads.
- retention, deletion, and security-contact policy for hosted metadata.
- clear response wording that the result is static pre-use assessment.

## Consequences

The PlayMCP product path becomes understandable for non-technical users: they add
one hosted MCP and ask whether another MCP should be connected. Runtime
enforcement remains available only in local or managed Gateway deployments where
target MCPs are actually behind the Gateway.

This creates an implementation gap: the current stdio upstream server is not
enough for PlayMCP registration. A future patch must add the inbound HTTP server
and public tool allowlist before public submission.
